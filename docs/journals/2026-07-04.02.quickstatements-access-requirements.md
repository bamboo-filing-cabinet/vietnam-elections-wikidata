# QuickStatements access requirements & bot-flag policy

Investigated 2026-07-04. Distinguishes claims **directly verified** by fetching
the source page (verbatim quotes) from claims drawn from **search synthesis**
of secondary sources.

## Bottom line

To run automated edits via QuickStatements you need to be an **autoconfirmed
user**, which the user's reading is correct about: **≥ 50 edits AND first edit
≥ 4 days old**. There is **no hard numerical threshold** in policy for when a
bulk run additionally needs bot approval — the rule is qualitative ("very large
or potentially-controversial runs").

## Autoconfirmed requirement (directly verified)

From [Wikidata:Autoconfirmed users](https://www.wikidata.org/wiki/Wikidata:Autoconfirmed_users),
verbatim: autoconfirmed status requires **"at least 50 edits and their first
edit is at least four days old."** The page adds that "precise requirements for
autoconfirmed status vary according to circumstances."

Autoconfirmed status is what grants the ability to "use certain tools for mass
editing of Wikidata" (also: edit semi-protected pages, skip most CAPTCHAs, have
edits auto-patrolled).

## QuickStatements requires autoconfirmed (directly verified)

- [QuickStatements 3.0 User guide](https://meta.wikimedia.org/wiki/QuickStatements_3.0/Documentation/User_guide),
  verbatim: **"You must be an autoconfirmed user in order to create and run
  batches in QuickStatements 3.0."**
- [Help:QuickStatements](https://www.wikidata.org/wiki/Help:QuickStatements),
  verbatim: **"an account needs to be autoconfirmed"**; no edit-rate throttle is
  stated in the documentation.
- Login is via **OAuth** — the 3.0 guide says to "Make sure you are logged into
  OAuth and your name is visible in the upper-right corner."

## When does a run additionally need bot approval? (directly verified quote)

[Help:QuickStatements](https://www.wikidata.org/wiki/Help:QuickStatements),
verbatim: **"Very large runs or potentially-controversial runs should go
through the approval process described in Wikidata:Bots."**

This is a **qualitative** trigger, not a numeric one. The
[Wikidata:Bots](https://www.wikidata.org/wiki/Wikidata:Bots) page I fetched did
not state any numeric edit-count / rate / batch-size threshold in the retrieved
sections.

## Bot-flag process, if a run qualifies (search synthesis — not fetched verbatim)

From search across
[Wikidata:Bots](https://www.wikidata.org/wiki/Wikidata:Bots),
[Wikidata:Requests for permissions/Bot](https://www.wikidata.org/wiki/Wikidata:Requests_for_permissions/Bot),
and [Wikidata talk:Bots](https://www.wikidata.org/wiki/Wikidata_talk:Bots):

- A bot flag requires a **separate account** — it is not applied to a personal
  account.
- File a request at **Wikidata:Requests for permissions/Bot** describing the
  task, then do a **test run of 50–250 edits** for community observation (the
  request must be opened before the test run).
- A community-cited rule of thumb: if you review every edit, it's not a bot; if
  you generate edits without checking them all, it is. The policy has **no
  well-defined numeric threshold** for QuickStatements mass edits — a gap the
  community itself acknowledges.
- **Flood flag** is a middle ground: for repetitive human edits, granted per
  request at the Bureaucrats' noticeboard, self-removable when done. Hides edits
  from RecentChanges/watchlists like a bot flag but is temporary.

## Batch-size practicality (directly verified)

Help:QuickStatements notes "batches with more statements do not load properly,"
with a practical ceiling around **~25,000 statements per batch** for simple
operations and **significantly lower** for statements with qualifiers or
references. The QS 3.0 guide says it has been "tested with over 140 thousand
commands." Our NA15 job (~500 persons × several qualified statements each) is
well within these limits, likely as one or a few batches.

## What this means for the plan

- **Prerequisite for the operator:** reach autoconfirmed first — 50 edits over
  ≥ 4 days on a normal Wikidata account before QuickStatements batches will run.
- **Bot approval (plan open question):** partially resolved. There is **no
  numeric threshold** forcing bot approval; ~500 curated, individually-reviewable
  person creations plausibly fit "not a bot" (each edit reviewed). But 500 new
  items is arguably a "very large run," so the safe path is either (a) run under
  the personal autoconfirmed account after sample-checking prepared statements,
  or (b) request a **flood flag** for the run, or (c) if we automate without
  per-item review, register a bot account and go through RfP/Bot. Decision still
  needs a human call; this is a judgment about supervision, not a hard rule.
