# Entity & constituency modeling decisions

Written 2026-07-04. Grounds three design questions in the actual
`vietnam-elections` data and live Wikidata. Companion artifact:
`reference/elections.json` (cycle → verified QIDs).

## 1. People-per-election mapping is person-centric, not per-election

The `vietnam-elections` data already separates the identifiers we need
(verified in `public/data/elections/na15-2021/results.json`):

- `person_id` (e.g. `person-72fb61a58093`) — stable, person-level
- `candidate_entry_id` (e.g. `na15-2021-const-144-3`) — per election + constituency
- `constituency_id` (e.g. `na15-2021-const-144`) — cycle-prefixed, per election

Wikidata uses **one item (QID) per human**. So the reconciliation output maps
`person_id → QID` at the **person** level. A delegate who serves in both NA15
and NA16 has **one** QID carrying **two** `P39` (position held) statements, each
qualified by its own `P2937` legislative term. Per-election participation lives
in the qualifiers, not in a per-cycle mapping table.

Decision: `mappings/` keys on `person_id`, spans cycles, one row per person.

## 2. Elections already have Wikidata entities — record them, don't create them

Verified present on Wikidata (see `reference/elections.json`):

| Cycle | Election event | Legislative term |
| --- | --- | --- |
| na15-2021 | `Q55718923` | `Q107979136` |
| na16-2026 | `Q106959840` | `Q138968785` |

The full election-event series exists back to 1976. The 2021 event
(`Q55718923`) already carries some `P991` (successful candidate, 3) and `P726`
(candidate, 2) links, plus `office contested`, `follows`/`followed by`,
`ballots cast`, etc. — so partial person↔election linkage already exists and
must be reconciled against.

The gap is in **our** dataset: it records none of these QIDs. `reference/
elections.json` is the small, stable, first artifact closing that gap. It is a
handful of rows — no bulk tooling needed — and everything downstream (the
`P39` term qualifier, `P2715` elected in) hangs off it.

## 3. Constituencies: do not model them as persistent Wikidata items

Our NA15 data models constituencies precisely (verified in
`constituencies.json`): **186 electoral units** ("đơn vị bầu cử"), **500 seats**
(500 winners in the data), each unit a set of districts tied to one of **63
provinces**. These units are **inherently per-election** — IDs are
cycle-prefixed and the districts composing each are redrawn every cycle by
population.

This is not hypothetical. Between the two cycles we track:

- **12 June 2025**: Vietnam merged provinces **63 → 34** (28 provinces + 6
  cities), effective **1 July 2025**.
- **December 2025**: the ~500 NA seats were reapportioned to the new 34 units
  for NA16-2026. Only 11 provinces were unchanged (Hà Nội, Huế, Lai Châu, Điện
  Biên, Sơn La, Cao Bằng, Lạng Sơn, Quảng Ninh, Thanh Hóa, Nghệ An, Hà Tĩnh).

So between NA15 and NA16 **both provinces and electoral units changed
wholesale**.

Decisions:

- ~~**Do not** mint Wikidata items for "đơn vị bầu cử" units — they are
  per-election drawings, not stable entities.~~ **Reversed 2026-07-06**: we now
  create per-election, cycle-scoped unit items for precise `P768` values. Being
  per-election drawings just means each cycle gets its own items. See
  `2026-07-06.03.electoral-unit-modeling.md`.
- For `P768` (electoral district), point at the **province item as it existed
  for that election's term**, matching the existing Wikidata convention (the 6
  districts currently in use are all province/city items — see
  `2026-07-04.01...`). Province-level is the ceiling.
- Keep the fine-grained electoral-unit composition **in `vietnam-elections`**,
  which already models it richly. Wikidata is not the source of truth for
  constituency composition.
- For NA16, `P768` targets must use the **post-merger** province items, which
  differ from NA15's.
